# Rust ↔ tatara-lisp mutual-generation macros

> **The user, 2026-04-26**: *"Lisp and Rust definitely should have a
> suite of easy macros or anything to be able to generate each other."*
>
> Sister doc to [`RUST-LISP-EMBEDDING.md`](RUST-LISP-EMBEDDING.md):
> embedding is about *runtime hosting*; this is about *compile-time
> generation* of the bridges between the two languages. Four
> quadrants. Each gets a macro. The full suite makes it mechanical to
> turn any Rust type into a Lisp authoring surface, any Lisp
> definition into a Rust type, and any function in either language
> into a callable in the other.

---

## I. The four-quadrant matrix

```
                      OUTPUT
                Rust              Lisp
              ┌──────────────────┬──────────────────┐
              │                  │                  │
        Rust  │   I.    AUTO     │  II.   #[derive  │
              │      (compiler)  │   (TataraDomain)]│  ← shipped
INPUT         │                  │      Rust→Lisp   │
              ├──────────────────┼──────────────────┤
              │                  │                  │
        Lisp  │  III.  defrust   │   IV.   AUTO     │
              │      Lisp→Rust   │      (reader)    │
              │       (NEW)      │                  │
              └──────────────────┴──────────────────┘
```

- **Quadrant I (Rust→Rust):** the Rust compiler. Not our concern.
- **Quadrant II (Rust→Lisp):** `#[derive(TataraDomain)]` already
  ships in `tatara-lisp-derive`. **Direction A → B done.**
- **Quadrant III (Lisp→Rust):** `(defrust ...)` macro that emits a
  Rust struct + `impl Deserialize` + `impl IntoLispValue`. **NEW —
  this document specifies it.**
- **Quadrant IV (Lisp→Lisp):** the tatara-lisp reader + macroexpander.
  Already shipped.

The four-corner matrix is closed once Quadrant III lands. Then **any
typed entity in either language has a macro that generates the
matching entity in the other.**

## II. Quadrant II — `#[derive(TataraDomain)]` (shipped)

The Rust→Lisp direction. Already in
[`tatara-lisp-derive`](https://github.com/pleme-io/tatara-lisp/tree/main/tatara-lisp-derive):

```rust
use tatara_lisp_derive::TataraDomain;

#[derive(TataraDomain, Serialize, Deserialize, Debug)]
#[tatara(keyword = "defmonitor")]
pub struct MonitorSpec {
    pub name: String,
    pub query: String,
    pub threshold: f64,
    pub window_seconds: Option<i64>,
}
```

Authored from Lisp:

```lisp
(defmonitor :name "prom-up"
            :query "up{…}"
            :threshold 0.99
            :window-seconds 300)
```

**6 lines of Rust ceremony** = one new Lisp authoring surface. The
proc macro generates the keyword extraction, type coercion, and
error reporting; the host registers the domain via
`tatara_lisp::domain::register::<MonitorSpec>()` at startup.

### II.1 Coverage today

Per the parallel research on `tatara-lisp-derive`'s source
(generation 1 supports):

- `String`, `Option<String>`, `Vec<String>`
- `i64`, `Option<i64>`
- `f64`, `Option<f64>`
- `bool`, `Option<bool>`
- Nested `#[derive(TataraDomain)]` structs

**Gap:** `HashMap<K, V>`, `enum` variants, `Vec<NestedStruct>`,
custom validation hooks. Mechanical extension.

## III. Quadrant III — `(defrust ...)` (NEW spec)

The Lisp→Rust direction. Specified here; implementation TBD as
Phase B work.

### III.1 The shape

```lisp
(defrust MonitorSpec
  :keyword     "defmonitor"
  :derives     [Serialize Deserialize Debug Clone]
  :fields
    {:name           :string
     :query          :string
     :threshold      :float
     :window-seconds (:optional :int)
     :tags           (:list :string)})
```

Generates the Rust source file:

```rust
// auto-generated by tatara-codegen/src/defrust.tlisp — do not hand-edit.
// Source: <github URL of the .tlisp that called defrust>

use serde::{Serialize, Deserialize};
use tatara_lisp_derive::TataraDomain;

#[derive(TataraDomain, Serialize, Deserialize, Debug, Clone)]
#[tatara(keyword = "defmonitor")]
pub struct MonitorSpec {
    pub name:           String,
    pub query:          String,
    pub threshold:      f64,
    pub window_seconds: Option<i64>,
    pub tags:           Vec<String>,
}

impl MonitorSpec {
    /// Convert this typed value into a tatara-lisp Value.
    pub fn into_lisp(&self) -> tatara_lisp_eval::Value {
        // Generated by defrust — produces an alist representation.
        ...
    }

    /// Construct from a tatara-lisp Value.
    pub fn from_lisp(v: &tatara_lisp_eval::Value) -> Result<Self, String> {
        ...
    }
}
```

Plus a matching `Cargo.toml` member entry, doctest, and (optionally)
JSON Schema export.

### III.2 Authored where

`(defrust ...)` is itself a tatara-lisp macro, lives in
[`pleme-io/tatara-codegen`](https://github.com/pleme-io/theory/blob/main/TATARA-CODEGEN.md)
under `lib/defrust.tlisp`. It takes the Sexp input, walks the
`:fields` map, emits Rust source as a string, and writes it to the
target crate's `src/<name>.rs`. Ships as part of `tatara-codegen`'s
catalog of code-generators.

### III.3 The bidirectional contract

When both `#[derive(TataraDomain)]` and `(defrust ...)` cover the
same shape, **the round-trip is identity:**

```
Rust struct ──[#[derive(TataraDomain)]]──▶ Lisp authoring → Sexp value
                                                                │
                                                                ▼
                                                          parse Sexp
                                                                │
                                                                ▼
            (defrust)──▶ regenerate Rust struct
                                ║
                                ║ identical bytes
                                ║
                            Rust struct
```

Modifying the source in either language regenerates the other.
**The two languages stay structurally synchronized by macro.**

## IV. Quadrant V — function-level bridges (the runtime side)

Beyond data types, **functions** in either language need bridges.
Two macros + one runtime primitive cover this:

### IV.1 `#[lisp_callable]` — Rust function → Lisp primitive

```rust
use tatara_lisp_script::lisp_callable;

#[lisp_callable(name = "blake3-hash", arity = 1)]
pub fn blake3_hash(input: String) -> String {
    blake3::hash(input.as_bytes()).to_hex().to_string()
}
```

Generates the `register_fn` boilerplate that `tatara-lisp-eval`
expects. Authors write the Rust function naturally; the macro
handles `Value` extraction + `Value` packing + error mapping.

**Status:** the `register_typed{N}()` helpers in `tatara-lisp-eval::ffi`
already do this manually. The proc macro is the next layer of
sugar — collapses 5 lines of boilerplate per function to one
attribute.

### IV.2 `(lisp-callable ...)` — Lisp function → Rust callable

```lisp
;; In a .tlisp:
(lisp-callable my-handler
  :rust-name "MyHandler"
  :args [(:request :Request)]
  :returns :Response

  (lambda (req)
    (response 200 (json-stringify (list (list "ok" true))))))
```

Generates a Rust trait impl + a thin `#[no_mangle] extern "C"`
wrapper if needed:

```rust
// auto-generated.
pub struct MyHandler;
impl LispCallable for MyHandler {
    fn invoke(&self, req: Request) -> Response {
        let interp = global_interpreter();
        let result = interp.eval_program("(my-handler request)", ...).unwrap();
        Response::from_lisp(&result).unwrap()
    }
}
```

Same content-pin model as `(defrust ...)` — emit the Rust file,
commit it to a peer crate, build it, ship the image.

### IV.3 Foreign-pointer bridges via `Value::Foreign`

For zero-copy interop where data shouldn't round-trip through
serialization, both languages have an opaque-pointer primitive:

- **Rust side**: `Value::Foreign(Arc<dyn Any + Send + Sync>)`
  already exists in `tatara-lisp-eval::value`.
- **Lisp side**: a Foreign value can be passed back to Rust via any
  registered fn that accepts it; the receiving Rust code downcasts
  to its concrete type.

Used for: file handles, database connections, large buffers, GPU
textures, anything where serialization would be the bottleneck.

## V. The full suite — all entries

Combining the data-type and function-level macros gives the canonical
set:

| Direction | Source | Macro | Generated artifact |
|---|---|---|---|
| Rust struct → Lisp domain | `MyStruct` | `#[derive(TataraDomain)]` | `(defmydomain ...)` form |
| Lisp domain → Rust struct | `(defrust ...)` | `(defrust ...)` macro | Rust source file |
| Rust function → Lisp primitive | `fn foo(x: T) -> U` | `#[lisp_callable]` | `register_fn(...)` call |
| Lisp function → Rust callable | `(lisp-callable ...)` | macro | Rust trait impl + extern wrapper |
| Rust enum → Lisp keyword set | `enum Shape` | `#[derive(LispKeyword)]` | `(:circle :square :triangle)` |
| Lisp set → Rust enum | `(defenum ...)` | macro | Rust enum source |
| Rust trait → Lisp protocol | `trait Foo` | `#[lisp_protocol]` | `(define-protocol foo ...)` |
| Lisp protocol → Rust trait | `(define-protocol ...)` | macro | Rust trait source |
| Rust value → Lisp value (runtime) | `Foreign(Arc<T>)` | `Value::Foreign` | n/a (already a value) |
| Rust function call from Lisp (runtime) | wasm component import | `wasm/load-component` | n/a (runtime resolves) |

Ten entries. The two unimplemented ones (`#[derive(LispKeyword)]` and
`#[lisp_protocol]`) follow the same `proc_macro` pattern as
`#[derive(TataraDomain)]`. Mechanical extensions of the existing
crate.

## VI. Why this matters operationally

Without this suite, every cross-language boundary requires hand-
written marshalling. With it:

- **Adding a new typed concept**: pick the home language, write the
  type once, generate the bridge.
- **Refactoring across languages**: change the source; regenerate;
  the other side stays in sync.
- **Type-level CI**: `tatara-codegen verify` (planned) re-runs every
  generation step and fails if the committed Rust/Lisp drift from
  what should-be-generated. Drift = bug.
- **Onboarding**: a new contributor learns one set of macros + the
  data-shape rules. No "how to bridge X → Y" cookbook entries.

The economic argument: in a 10-language polyglot project the bridge
maintenance is N²; here it's effectively N×1, where the "1" is the
mutual-generation suite this document specifies.

## VII. Phasing

- **Phase A (designed — this doc):** the four-quadrant matrix +
  the ten-entry suite + behavior contracts.
- **Phase B (already in place):** Quadrant II (`#[derive(TataraDomain)]`)
  + Quadrant IV (the reader/expander).
- **Phase C (NEXT):**
  1. `(defrust ...)` as a `tatara-codegen/lib/defrust.tlisp` macro that
     emits Rust source. ~200 lines of tatara-lisp.
  2. `#[lisp_callable]` proc macro in `tatara-lisp-derive`. ~150 lines
     of Rust syn-quote work.
- **Phase D:**
  1. `#[derive(LispKeyword)]` for Rust enums.
  2. `(defenum ...)` for the reverse direction.
- **Phase E:**
  1. `#[lisp_protocol]` for Rust traits.
  2. `(define-protocol ...)` for the reverse.
- **Phase F:**
  1. `tatara-codegen verify` — drift detection across the suite.

Total estimated work to complete Phase C-F: ~1-2 weeks of focused
implementation. Each phase is independent; ship in order.

## VIII. The closing principle

**Every typed entity in either language has a one-attribute or
one-form macro that generates the matching entity in the other.**

That sentence is the test for "is the Rust/Lisp boundary mechanical?"
Today it's mechanical for one direction (Rust → Lisp via
`TataraDomain`). After Phase C-F it's mechanical for all four
quadrants + function-level bridges + enum/trait shape coverage.

The user's stated long-term goal — *Rust hosts Lisp; Lisp hosts
Rust; the boundary is selection, not barrier* — needs both the
runtime embedding ([`RUST-LISP-EMBEDDING.md`](RUST-LISP-EMBEDDING.md))
AND the compile-time mutual generation (this doc). Together they
close the loop:

```
Runtime:  embed each other (RUST-LISP-EMBEDDING)
Compile:  generate each other (RUST-LISP-MUTUAL-MACROS — this doc)

Result:   the languages compose at every level. Type definitions
          travel; function calls travel; runtime instances travel.
          The boundary disappears.
```
