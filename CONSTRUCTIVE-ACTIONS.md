# Constructive Actions — pleme-io's GitHub Actions publishing pattern

> **Status:** canonical (2026-04-27). Implementation tracked in [`PLEME-ACTIONS-PLAN.md`](./PLEME-ACTIONS-PLAN.md).
>
> This document specifies the pattern pleme-io uses to author, build, version,
> publish, and consume reusable GitHub Actions across the fleet (pleme-io
> internal, sibling orgs, customers, public consumers). It complements
> [`THEORY.md`](./THEORY.md) §VI (typescape rendering) and §II.4
> (Rust + Lisp pattern).

---

## The frame

A "GitHub Action" is a versioned unit of execution invoked from a workflow:

```yaml
# Consumer's .github/workflows/deploy.yml
- uses: pleme-io/terragrunt-apply@v1
  with:
    leaf: saas/terraform/.../arc-controller
    action: apply
```

The published surface — the `org/repo@ref` that consumers paste — is what we
must keep stable. Everything behind it (Rust source, build pipeline,
attestation chain, release machinery) is the pleme-io substrate's job.

---

## Why not the helmworks pattern (one repo, many artifacts)

Helmworks publishes ~50 charts to one OCI registry namespace. That works
because **OCI is the natural distribution surface for Helm charts** —
consumers reference a chart by name + version, the registry owns the
publishing namespace.

GitHub Actions has a **different idiomatic distribution surface**:
`uses: org/repo@ref`. Consumers expect:

- One repo per action (clean marketplace listing)
- Semver tags (`v1.2.3`) on each repo
- Moving major-version pointers (`v1` always tracks the latest 1.x.y)
- Repo-level pinning to SHAs for supply-chain security

Trying to consume a monorepo's action via `pleme-io/pleme-actions/terragrunt-apply@v1`
fights GitHub's UX, loses independent semver, and binds the version of one
action to every other action in the repo. No major action publisher
(`actions/`, `aws-actions/`, `docker/`, `octokit/`, etc.) does it that way.

The right inversion: **one catalog, many published repos.** Same compounding
philosophy as helmworks — single source of truth, mechanical generation —
but the rendered artifacts are separate repos, each with their own semver,
marketplace listing, and pinning surface.

## Architecture

### Typescape-render-first

Every published action is, in the substrate's eyes, a **typed value in
arch-synthesizer's `Action` domain**. The repo on GitHub, its Rust crate,
its action.yml, its release workflow — all of it — is rendered from that
typed value. The Rust struct is the source of truth; the rendered files
are projections.

Per the pleme-io Rust + Lisp pattern (`theory/THEORY.md` §II.4), the
`Action` typed Rust struct in arch-synthesizer derives `TataraDomain`,
which auto-generates the tatara-lisp authoring surface:

```rust
// arch-synthesizer/src/action_domain.rs
#[derive(TataraDomain, Serialize, Deserialize, Debug, Clone)]
#[tatara(keyword = "defaction")]
pub struct Action {
    pub name: String,
    pub description: String,
    pub inputs: Vec<ActionInput>,
    pub outputs: Vec<ActionOutput>,
    pub behavior: ActionBehavior,
    pub semver_compat: SemverPolicy,
    pub attestation: AttestationPolicy,
}
```

Authoring a new action is **six lines of ceremony** plus a `(defaction …)`
form in `pleme-actions-catalog.lisp`. The Lisp form parses to the typed
Rust value; arch-synthesizer renders the value to the per-action repo
artifacts; repo-forge writes them to disk; substrate builds the binary;
forge tags the release. Every step is mechanical + idempotent.

### Layers

| Layer | Where | Purpose |
|---|---|---|
| **Action domain (typescape)** | `arch-synthesizer/src/action_domain.rs` | The typed Rust struct that defines what an action IS. Source of truth. |
| **Tatara-lisp surface** | `repo-forge/pleme-actions-catalog.lisp` | `(defaction …)` forms, one per action. Auto-generated parser via `#[derive(TataraDomain)]`. |
| **Rendering pipeline** | `arch-synthesizer/src/action_domain/render.rs` | Each `Action` value renders to the full set of per-action-repo artifacts (Cargo.toml, action.yml, flake.nix, release.yml, README, repo metadata). |
| **Archetype** | `repo-forge/archetypes/rust-action/` | Templates the rendering pipeline writes through. Boilerplate-vs-authored discipline (per `repo-forge/docs/principles.md`). |
| **Substrate primitive** | `substrate/lib/rust-action-release-flake.nix` | Cross-compiles the Rust binary to all four targets, releases to GitHub, emits the action.yml binary-download glue. |
| **Shared crate** | `pleme-io/pleme-actions-shared` | Rust toolkit shim — input/output parsing per the GitHub Actions toolkit conventions, step-summary builders, structured logging. Every action depends on this. |
| **Per-action repo** | `pleme-io/<action-name>` | Generated by the rendering pipeline. Houses the Rust crate, action.yml, release workflow, tests. Boilerplate is rendered (managed); `src/` is operator-authored. |
| **Index repo** | `pleme-io/pleme-actions` | Pure-docs hub — README catalogs every published action with its purpose, version compat matrix, and migration notes. No code. |
| **Consumers** | Anywhere | Reference via `uses: pleme-io/<action>@v1`. |

### The flow

```
catalog edit
   │
   ▼
repo-forge migrate ── renders or updates per-action repo boilerplate
   │
   ▼
git push (per-action repo)
   │
   ▼
release.yml workflow ── nix run .#release builds binaries via substrate,
                        forge attests them, GH release tagged vX.Y.Z,
                        moving major tag vX advanced
   │
   ▼
consumer picks it up via uses: org/repo@v1
```

A new action: one catalog entry + one `repo-forge new` invocation produces
the entire repo, ready to commit + tag + release.

A new version of an existing action: bump version in catalog (or directly
in the per-action repo's Cargo.toml) + push + release workflow tags +
publishes.

### What lives in each per-action repo

Generated boilerplate (managed by repo-forge — never hand-edit):

```
.github/workflows/
  release.yml          ← on tag push, runs `nix run .#release` via forge
  ci.yml               ← cargo check + clippy + test on every PR
flake.nix              ← consumes substrate's rust-action-release-flake
Cargo.toml             ← package manifest, depends on pleme-actions-shared
action.yml             ← composite action wrapper that downloads the binary
README.md              ← inputs/outputs table, examples, version compat
LICENSE                ← MIT (default for pleme-io OSS)
.envrc                 ← use flake
.gitignore             ← target/, result, etc.
```

Authored content (operator owns):

```
src/                   ← the Rust binary's logic
src/main.rs            ← wires inputs from `pleme_actions_shared::Input`
                         to the action's behavior, emits outputs via
                         `pleme_actions_shared::Output`
tests/                 ← integration tests + unit tests
docs/                  ← optional deeper docs beyond README
```

The Rust crate is small by design — each action does one thing well. Shared
concerns (parsing, logging, attestation injection, error handling) live in
`pleme-actions-shared`.

### Versioning + tags

Per-action semver, GitHub Actions convention:

- `v1.2.3` — exact version, immutable
- `v1.2` — moves with the latest 1.2.x release
- `v1` — moves with the latest 1.x.y release
- `main` — never use as a consumer ref (security risk + breakage risk)

Release workflow auto-updates the moving major-version tag on every release.
Operators can freeze on `v1.2.3` for max stability or track `v1` for
auto-applied patches.

Breaking changes bump the major: `v1` → `v2`. Both major tags exist
simultaneously (no auto-migration); consumers pick when to upgrade.

## Why this fits the pleme-io substrate

| Existing substrate piece | Role in actions publishing |
|---|---|
| **substrate** (`rust-tool-release-flake.nix`, `rust-workspace-release-flake.nix`) | Already cross-compiles to all 4 targets and releases to GitHub. The new `rust-action-release-flake.nix` extends this with action.yml emission + binary-download glue. Zero new build infrastructure. |
| **forge** (`forge release …`) | Action repos hook the forge release flow — same cross-platform build, same attestation chain (sekiban hashes injected into the GH release), same retry/idempotency. Adds `forge action release` as a thin convenience wrapper. |
| **repo-forge** (`(defrepo …)`, `(defarchetype …)`) | The action archetype is just another archetype. Catalog entries in `pleme-actions-catalog.lisp` follow the same `(defaction "name" …)` shape that's already idiomatic. `repo-forge new --archetype action` and `repo-forge migrate` handle scaffolding + drift. |
| **arch-synthesizer (typescape)** | Each catalog entry compiles to a typed Rust value. Wrong combinations (e.g., missing required input, conflicting type signatures) fail at compile time, not at release. |
| **forge-gen** | If the action's behavior is itself driven by an OpenAPI spec or similar (e.g., a GitHub-API-wrapping action), forge-gen can generate the Rust client; the action becomes a thin wrapper around the generated client. |
| **tameshi attestation** | Each action release includes the binary's BLAKE3 hash + the source SHA + the build attestation. Consumers can verify they're getting exactly the binary from the documented source commit. |

The new pieces — `rust-action-release-flake.nix`, the `(defaction-archetype …)`,
the catalog format, `pleme-actions-shared` — are the *only* new substrate
additions. Everything else reuses what already exists.

## Why this is "constructive"

Following the [Constructive Substrate Engineering](https://github.com/pleme-io/theory/blob/main/CONSTRUCTIVE-SUBSTRATE-ENGINEERING.md)
discipline:

- **Knowledge proven by construction.** The Rust compiler refuses to build
  an action whose declared inputs don't match the typed `Input` struct.
  An action that compiles cannot have undeclared inputs, mismatched types,
  or missing outputs.
- **Mechanical generation.** A new action is one catalog entry — repo-forge
  produces the rest. No handcrafted YAML + Rust + Nix boilerplate that
  drifts.
- **Versioned promises.** Every action's semver is a promise about its
  inputs/outputs/behavior. Breaking promises forces a major bump; the
  catalog enforces this with type signatures.
- **Compounding leverage.** The marginal cost of a new action is one
  catalog entry. The substrate handles the rest. Three actions today
  becomes thirty actions next year without thirty times the work.

## Anti-patterns this design forbids

- **Hand-authored composite actions in pleme-io repos.** All pleme-io
  reusable workflows go through this pipeline; bespoke `action.yml` files
  with hand-written shell are drift.
- **Monorepo with subdirectory paths.** Consumers should never see
  `uses: pleme-io/pleme-actions/terragrunt-apply@v1`. Each action is its
  own repo.
- **Cross-action versioning coupling.** Action A's version must NOT depend
  on action B's version. They're independent published artifacts.
- **Container-image actions.** Docker container actions are slow to start
  and harder to attest. Composite + Rust binary download is the only
  approved shape. (Exception: if a consumer demands a container action
  for a specific reason, route it through the substrate explicitly.)
- **Bypassing the catalog.** A new pleme-io action that's not in
  `pleme-actions-catalog.lisp` is invisible to drift detection + version
  audits. Always catalog first.
- **Using `@main` in a consumer reference.** Always pin to a tag (`@v1` for
  auto-tracking, `@v1.2.3` or `@<sha>` for max stability).
- **Skipping the README + version-compat matrix.** Every action repo's
  README must list its semver compatibility (which inputs/outputs are
  guaranteed within `v1`, what changed between majors).

## Naming

- **Repo name** = action name = the binary name. Short, kebab-case,
  imperative or noun-first (`terragrunt-apply`, `k8s-pause-verify`,
  `helmworks-render-check`). Consumers paste this directly; brevity matters.
- **Index repo** = `pleme-actions` (singular hub for the plural set).
- **Shared crate** = `pleme-actions-shared` (clear it's library glue, not
  a published action).
- **No `action-` or `gha-` prefix** on individual action repos. The
  `pleme-io/<name>@v1` form already implies it's an action.

Following the [pleme-io naming convention](./THEORY.md#§II.3): primary
substrate primitives use Japanese (`tatara`, `shikumi`, etc.); concrete
infrastructure pieces with Brazilian-Portuguese flavor or English-descriptive
where it serves the consumer better. Action names are consumer-facing, so
**English-descriptive wins** — `terragrunt-apply` over a poetic alternative.

## Initial action set

Three concrete patterns currently inlined across pleme-io workflows that
become published actions in the first wave:

1. **`pleme-io/terragrunt-apply`** — typed `terragrunt apply` with
   sensitive `TF_VAR_*` from secrets, plan/apply/destroy mode toggle,
   structured plan output, post-apply summary. Replaces inline shell
   in any terragrunt-running CI (pleme-io's pangea workflows + any
   downstream consumer doing typed-IaC apply).

2. **`pleme-io/k8s-pause-verify`** — assert pause-contract invariants
   after a deploy: pod-count by label selector, AutoscalingRunnerSet
   status, ConfigMap presence. Used by any workflow that ships
   paused-first deployments (ARC runner pools, shihaisha managed
   services, future paused-deploy workloads).

3. **`pleme-io/helmworks-render-check`** — pull a chart by version, render
   with operator values, fail on contract violations (pause, attestation,
   schema). Pre-merge gate for any helmworks consumer.

Each pulls ~50–100 lines of duplicated YAML + bash out of N workflows.
Adding more actions is purely catalog-driven thereafter.

## Decision log

| Date | Decision | Rationale |
|---|---|---|
| 2026-04-27 | One repo per action, not monorepo | GitHub Actions UX (marketplace, semver, SHA pinning) demands per-repo. Monorepo loses all of that. |
| 2026-04-27 | Catalog-driven, mechanical generation | Same compounding philosophy as helmworks. Adding action N+1 stays cheap. |
| 2026-04-27 | Composite action.yml + Rust binary | Composite is fast (no Docker pull). Rust binary is typed + testable. Substrate already builds Rust binaries for all 4 platforms. |
| 2026-04-27 | English-descriptive names for action repos | Consumer-facing surface; readability + searchability over linguistic theme. |
| 2026-04-27 | Implement after at least 2 distinct production callers identify the pattern | Want 2 data points to design the action API correctly. With one caller we'd over-fit. |
