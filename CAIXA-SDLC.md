# Caixa SDLC — author to live, ruthlessly standardized

> **Frame.** [`META-FRAMEWORK.md`](./META-FRAMEWORK.md) describes the
> four-layer compute hierarchy and *where* compute lives. This document
> describes *how* compute moves between those layers in production —
> the GitHub-native SDLC every caixa author follows, end to end. No
> manual cluster ops; no console-clicking; no drift. The substrate
> primitives in `pleme-io/substrate/.github/workflows/` are load-bearing.

---

## I. The author surface

A caixa repo is a typed package. The author touches three things:

```
my-caixa/
├── caixa.lisp                              ← typed manifest (defcaixa …)
├── servicos/<nome>.computeunit.yaml        ← runtime contract (only Servicos)
├── lib/<nome>.lisp  | exe/  | src/         ← source (kind-dependent)
└── .github/workflows/release.yml           ← 5 lines (`uses: substrate/...`)
```

That's it. Everything below — Helm chart, FluxCD bundle, OCI artifact,
operator CR, KEDA scale-to-zero — is generated by a substrate
primitive the author never edits.

---

## II. The four reusable workflows

Each caixa picks **one** workflow to publish under, plus optionally
the validate + forge gates. All live in `pleme-io/substrate/.github/workflows/`.

| workflow | for | what it does | secrets | permissions |
|---|---|---|---|---|
| `caixa-publish.yml` | Servico (Rust → wasm) | feira lint → `nix build .#dockerImage` → ghcr push (`v<versao>` + `:latest`) → git tag | `inherit` | `contents: write`, `packages: write` |
| `caixa-publish-tlisp.yml` | pure-Lisp Servico / Biblioteca / Binario | feira lint → cse-lint --strict → git tag (no OCI image — runtime resolves via github: URL) | `inherit` | `contents: write` |
| `caixa-validate.yml` | monorepos / dev branches / PRs | cse-lint --strict + feira lint, optional `feira build` | `inherit` | `contents: read` |
| `caixa-forge.yml` | caixas wrapping external APIs | forge-gen against `spec/openapi.yaml` → auto-PR on drift | `inherit` | `contents: write`, `pull-requests: write` |

The caller workflow shape is identical for all four:

```yaml
name: release
on: { push: { branches: [main] }, workflow_dispatch: {} }
jobs:
  release:
    uses: pleme-io/substrate/.github/workflows/<workflow>.yml@main
    secrets: inherit
    permissions: { contents: write, packages: write }
```

## III. The full chain, pointer-by-pointer

```
edit caixa.lisp + source  ← Layer 0
   │
   │ git push main
   ▼
substrate/.github/workflows/caixa-publish.yml          ← Layer 1
  • cse-lint repo . --strict          (6 CSE invariants gate)
  • feira lint                         (manifest + Lisp parse)
  • nix build .#dockerImage           (substrate's wasi-service-flake)
  • skopeo push ghcr.io/pleme-io/<nome>:v<versao> + :latest
  • git tag v<versao> + push           (Zig-style pin)
   │
   ▼
[author runs] feira deploy --cluster <name> --apply    ← Layer 2 → 3 → 4 → 5
  • caixa-flux::programs_yaml_entry    (typed L2 → L3 fragment)
  • upsert spec.values.programs[]      (HelmRelease in k8s repo)
  • git commit + push                  (FluxCD repo)
   │
   ▼
FluxCD reconciles                                       ← Layer 5
  • lareira-fleet-programs HelmRelease re-renders
  • emits one wasm.pleme.io/v1alpha1 ComputeUnit per :programs entry
   │
   ▼
wasm-operator reconciles                                ← Layer 2 (runtime)
  • ensure_deployment / _service / _scaled_object
  • OwnerReference back to ComputeUnit (GC tracks lifecycle)
   │
   ▼
Pod runs wasmtime serve                                 ← Layer 1 (runtime)
  • wasm-engine pulls module.source (oci:// or github://)
  • exposes wasi:http/proxy on the configured port
  • KEDA HTTP scales 0→N→0 if breathability.enabled
```

The chain is *typed at every step*. A regression at Layer N can't
silently propagate to Layer N+1: the typed boundary refuses ill-formed
inputs.

## IV. The caixa-compliant Lisp substrate

A "caixa-compliant Lisp substrate" is any Kubernetes context (1+ nodes,
1+ clusters, 1+ containers) that hosts:

1. **`lareira-tatara-stack`** Helm chart (`pleme-io/lareira-charts`)
   — installs:
   - `wasm-operator` Deployment + RBAC
   - `wasm.pleme.io/v1alpha1 ComputeUnit` CRD
   - `wasm-engine` runtime base image pull policy
   - KEDA HTTP add-on (for breathability)

2. **`lareira-fleet-programs`** Helm chart (same repo)
   — the *aggregator* that takes `.Values.programs[]` (one entry per
   caixa Servico) and renders one `ComputeUnit` per entry. Single
   HelmRelease owns the entire fleet — no per-program HelmRelease
   sprawl.

3. **GitOps source** (any Flux-compatible repo, e.g.
   `pleme-io/k8s/clusters/<name>/programs/release.yaml`) — points at
   `lareira-fleet-programs` with the cluster's typed `programs:` array.

That's the entire substrate. One Helm chart for the runtime, one for
the fleet manifest, one GitOps repo. Scales from a single rio-style
homelab K3s node to a multi-region AWS EKS fleet without architectural
change — only the FluxCD source moves.

### Substrate composition guarantees

Because everything below `caixa.lisp` is generated by typed
primitives:

- **A new cluster is `helm install lareira-tatara-stack` + `helm install lareira-fleet-programs` + a `programs:` array.** No per-cluster bespoke wiring.
- **A new caixa is one `git push` + one `feira deploy`.** The caller
  never edits the cluster's runtime; the operator picks up the
  ComputeUnit and reconciles deterministically.
- **A new node** (extending an existing cluster) is `kubectl join`. The
  KEDA HTTPScaler distributes load across the node pool. The operator
  doesn't care which node hosts which Pod.
- **A new container shape** (bigger memory, GPU, FPGA) is a `resources:`
  override in the `programs:` entry. caixa-flux merges the override
  into the rendered ComputeUnit unchanged.

## V. Forge / substrate / tatara-lisp pattern compatibility

Every caixa composes the three load-bearing pleme-io patterns:

### Substrate (Nix)

The caixa's `flake.nix` consumes a substrate helper (one of):
- `wasi-service-flake.nix` — Rust source → wasm32-wasip2 component
- `wasi-service-flux-flake.nix` — same + FluxCD bundle emission
- `tatara/program-flake.nix` — `.tlisp` source → wasm via tlisp2nix
- `rust-library.nix` / `rust-tool-release-flake.nix` — for Bibliotecas / Binarios

A hand-rolled `pkgs.rustPlatform.buildRustPackage` in a caixa flake is
a **cse-lint hand-roll violation** and fails the gate.

### Tatara-lisp (authoring)

The caixa's `caixa.lisp` is itself a tatara-lisp form (`(defcaixa …)`).
Every `:bibliotecas` / `:exe` / `:servicos` slot points at typed sub-
forms parsed at build time by `caixa-core::Caixa::from_lisp`. Lisp-
shaped Servicos additionally have `main.tlisp` evaluated by tatara-
script (today) or compiled to wasm via tlisp2nix (when wasm-engine
matures).

### Forge (codegen)

A caixa wrapping an external API (Akeyless, Datadog, Cloudflare,
GitHub, etc.) ships an `spec/openapi.yaml` and adopts
`caixa-forge.yml`. The workflow runs `forge-gen` and PRs the
regenerated SDKs / MCP servers / IaC providers into the caixa.
The caixa author never edits generated code.

## VI. The cse-lint enforcement loop

Every caixa repo's CI runs `cse-lint repo . --strict` as the first
gate (via `caixa-publish.yml` / `caixa-publish-tlisp.yml` /
`caixa-validate.yml`). The six CSE invariants gate every push:

| invariant | check | violation triggered when |
|---|---|---|
| models stay current | `claude-md-pointer` | CLAUDE.md doesn't link to CSE doc |
| solve problems once | `hand-roll` | flake.nix doesn't consume a substrate helper |
| acquire and contextualize | `manifest-membership` | (workspace-level only) |
| idiom-first | `module-trio-adoption` | `// { homeManagerModules.default = … }` legacy pattern present |
| promises hold mechanically | `deployment-coverage` | (workspace-level only) |
| generation over composition | `caixa-naivete` | no `caixa.lisp` at the root |

A regression on any invariant fails CI before publish. The fleet
*structurally cannot* drift from the SDLC standardization.

## VII. Migrating an existing repo

For each pleme-io repo that's not yet caixa-native:

```bash
cd ~/code/github/pleme-io/<repo>

# 1. Author the manifest (or copy hello-rio's caixa.lisp + tweak)
nix run github:pleme-io/caixa#feira -- init <name> --path .   # scaffold

# 2. (Servico only) move existing computeunit.yaml under servicos/
mkdir -p servicos
git mv computeunit.yaml servicos/<name>.computeunit.yaml

# 3. Wire the right reusable workflow
mkdir -p .github/workflows
cat > .github/workflows/release.yml <<'EOF'
name: release
on: { push: { branches: [main] }, workflow_dispatch: {} }
jobs:
  release:
    uses: pleme-io/substrate/.github/workflows/caixa-publish.yml@main   # or -tlisp / -validate
    secrets: inherit
    permissions: { contents: write, packages: write }
EOF

# 4. Verify locally
nix run github:pleme-io/cse-lint -- repo . --strict
nix run github:pleme-io/caixa#feira -- chart                           # if Servico
nix run github:pleme-io/caixa#feira -- deploy --cluster <name> --dry-run

# 5. Push
git add caixa.lisp servicos/ .github/workflows/release.yml
git commit -m "ci: adopt substrate's caixa-publish reusable workflow"
git push origin main
```

CI now publishes (or validates) the repo on every commit. An unbroken
chain from `git push` to live ComputeUnit takes one extra step:
`feira deploy --cluster <name> --apply`.

## VIII. The standardization budget

The user-facing surface is bounded:

- **2 files** to author per caixa: `caixa.lisp` + (Servico only) `servicos/<name>.computeunit.yaml`
- **1 workflow** to wire: `.github/workflows/release.yml` (~10 lines)
- **1 verb** to deploy: `feira deploy --cluster <name> --apply`
- **1 verb** to ship: `git push`

Anything else — Helm templates, FluxCD GitRepository / HelmRelease /
Kustomization, ComputeUnit CR shape, OCI artifact format, KEDA
configuration, Pod spec, resource quotas, RBAC — is generated, gated,
and substrate-owned. Every layer of the chain is typed; a regression
on any layer is a build error, not a runtime surprise. That's the
operator-out-of-the-loop guarantee the substrate exists to provide.
